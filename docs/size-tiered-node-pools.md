# Size-Tiered Node Pools

## Problem

Every team gets one Karpenter NodePool (generated by Kyverno) with instance types ranging from xlarge to 12xlarge. Karpenter batches pending pods and provisions the largest instances available. Combined with `consolidationPolicy: WhenEmpty`, oversized nodes persist until fully drained.

Result: a pod requesting 1 CPU / 4 GiB can end up on a 12xlarge (48 vCPU / 384 GiB) that sits 95% idle until the job completes.

Data from data-engineering (30 days, 24,037 runs):

| Tier | Runs | % | Description |
|------|------|---|-------------|
| S | 8,052 | 33% | < 3 CPU, < 24 GiB |
| M | 8,676 | 36% | 3-14 CPU or 24-100 GiB |
| L | 6,308 | 26% | 14-30 CPU or 100-200 GiB |
| XL | 1,001 | 4% | 30+ CPU or 200+ GiB |

This pattern repeats across all teams (ml-platform: 177 nodes, foundational-models: 120 nodes, etc.).

## Solution

Split each team's single NodePool into four size-tiered pools. Flotilla adds an additional toleration based on the pod's resource requests to route it to the correct tier.

### Tier Definitions

| Tier | Instance Sizes | Max Allocatable CPU | Max Allocatable Memory | Taint Value |
|------|---------------|--------------------|-----------------------|-------------|
| S | large, xlarge | ~3300m | ~28 GiB (r.xlarge) | `flotilla-pool=s:NoSchedule` |
| M | 2xlarge, 4xlarge | ~15300m | ~119 GiB (r.4xlarge) | `flotilla-pool=m:NoSchedule` |
| L | 8xlarge | ~31300m | ~241 GiB (r.8xlarge) | `flotilla-pool=l:NoSchedule` |
| XL | 12xlarge | ~47300m | ~363 GiB (r.12xlarge) | `flotilla-pool=xl:NoSchedule` |

Allocatable accounts for kubelet systemReserved (200m CPU, 1.75 GiB), kubeReserved (200m CPU), eviction (5% memory), and daemonset overhead (~300m CPU, ~500 MiB).

### Instance Family Specs

All tiers allow `instance-category: [m, c, r]` with `instance-generation > 4`. Karpenter picks the cheapest family that fits the pod's requests.

| Size | vCPU | Memory (c) | Memory (m) | Memory (r) |
|------|------|-----------|-----------|-----------|
| large | 2 | 4 GiB | 8 GiB | 16 GiB |
| xlarge | 4 | 8 GiB | 16 GiB | 32 GiB |
| 2xlarge | 8 | 16 GiB | 32 GiB | 64 GiB |
| 4xlarge | 16 | 32 GiB | 64 GiB | 128 GiB |
| 8xlarge | 32 | 64 GiB | 128 GiB | 256 GiB |
| 12xlarge | 48 | 96 GiB | 192 GiB | 384 GiB |

Memory ratios: c=2 GiB/vCPU, m=4, r=8. Consistent across gen 5/6/7.

### Routing Thresholds (flotilla-os)

```go
func poolTier(cpuMillis int64, memoryMiB int64) string {
    switch {
    case cpuMillis >= 30000 || memoryMiB >= 200000:
        return "xl"
    case cpuMillis >= 14000 || memoryMiB >= 100000:
        return "l"
    case cpuMillis >= 3000 || memoryMiB >= 24000:
        return "m"
    default:
        return "s"
    }
}
```

Threshold rationale:
- **S→M (3000m / 24 GiB):** xlarge allocatable is ~3300m CPU / ~28 GiB (r.xlarge). Buffer prevents stuck pods.
- **M→L (14000m / 100 GiB):** 4xlarge allocatable is ~15300m / ~119 GiB (r.4xlarge).
- **L→XL (30000m / 200 GiB):** 8xlarge allocatable is ~31300m / ~241 GiB (r.8xlarge).

### NodePool Requirements (all tiers)

```yaml
requirements:
  - key: karpenter.k8s.aws/instance-category
    operator: In
    values: ["m", "c", "r"]
  - key: karpenter.k8s.aws/instance-generation
    operator: Gt
    values: ["4"]
  - key: karpenter.k8s.aws/instance-size
    operator: In
    values: [...]  # per tier: [large, xlarge] / [2xlarge, 4xlarge] / [8xlarge] / [12xlarge]
  - key: topology.kubernetes.io/zone
    operator: In
    values: ["us-east-1b"]
  - key: kubernetes.io/arch
    operator: In
    values: ["amd64"]
  - key: karpenter.sh/capacity-type
    operator: In
    values: ["spot", "on-demand"]
```

### Labels and Taints (per node)

Each node gets labels:
- `team: <team>` (existing)
- `flotilla-pool: <s|m|l|xl>` (new — used by pod nodeAffinity to positively select the tier)

Each node gets TWO taints:
1. `<team>=true:NoSchedule` (existing — team isolation for cost attribution)
2. `flotilla-pool=<s|m|l|xl>:NoSchedule` (new — size routing)

A pod needs both **tolerations** (to get past the taints) and **nodeAffinity** (to target the specific tier). Tolerations alone are permissive — without affinity a pod could land on old pools or default nodes that lack the taint.

## Changes Required

### 1. Kyverno Policy (eng-infrastructure)

New policy `generate-team-tiered-pools` generates 4 NodePools + 4 EC2NodeClasses per team:
- `<team>-s` — instance-size: [large, xlarge]
- `<team>-m` — instance-size: [2xlarge, 4xlarge]
- `<team>-l` — instance-size: [8xlarge]
- `<team>-xl` — instance-size: [12xlarge]

Each NodePool template includes:

```yaml
template:
  metadata:
    labels:
      team: "{{request.object.metadata.labels.team}}"
      flotilla-pool: "<s|m|l|xl>"
  spec:
    taints:
      - key: "{{request.object.metadata.labels.team}}"
        value: "true"
        effect: NoSchedule
      - key: "flotilla-pool"
        value: "<s|m|l|xl>"
        effect: NoSchedule
```

The `flotilla-pool` label is what pod nodeAffinity matches against. The taint prevents unprepared pods from landing there.

Drop the staging/prod split — one set of pools per team. Environment label stays on the node for observability but isn't a scheduling constraint.

### 2. Flotilla Code — EKS Adapter (flotilla-os)

`execution/adapter/eks_adapter.go` — `constructTolerations()`:

```go
// Existing team toleration (unchanged)
tolerations = append(tolerations, corev1.Toleration{
    Key:      team,
    Operator: "Equal",
    Value:    "true",
    Effect:   "NoSchedule",
})

// New: size-tier toleration
tier := poolTier(cpuMillis, memoryMiB)
tolerations = append(tolerations, corev1.Toleration{
    Key:      "flotilla-pool",
    Operator: "Equal",
    Value:    tier,
    Effect:   "NoSchedule",
})
```

`constructAffinity()` — add node selector term for `flotilla-pool` label:

```go
nodeSelectorTerms[0].MatchExpressions = append(
    nodeSelectorTerms[0].MatchExpressions,
    corev1.NodeSelectorRequirement{
        Key:      "flotilla-pool",
        Operator: corev1.NodeSelectorOpIn,
        Values:   []string{tier},
    },
)
```

This ensures the pod ONLY schedules on nodes in the correct tier — tolerations permit, affinity selects.

### 3. Flotilla Code — EMR Engine (flotilla-os)

`execution/engine/emr_engine.go` — same pattern for Spark driver/executor pods. Both `constructTolerations()` and `constructAffinity()` get the tier-based toleration and affinity.

### 4. Out of Scope

- **wait_for_data pods** — Kyverno mutates to `t3.xlarge` nodeSelector, lands on default pool
- **job submitter pods** — Kyverno mutates to `t3a.2xlarge`, lands on default pool
- **GPU pools** — separate taint (`nvidia.com/gpu`), different instance families

## Backwards Compatibility

1. Deploy new Kyverno policy → new tiered pools created alongside existing ones
2. Existing pods (no `flotilla-pool` toleration) stay on old pools — they can't schedule on new pools (missing toleration)
3. Deploy flotilla code → new pods get both tolerations → land on new tiered pools
4. Old pools drain naturally as running jobs complete
5. Remove old Kyverno policy once old pools are empty

No downtime. No migration of running jobs.

## Expected Impact

- 69% of runs (S+M tiers) constrained to max 4xlarge instead of 12xlarge
- Nodes drain faster (smaller node = fewer jobs sharing = empties sooner)
- `WhenEmpty` consolidation becomes effective on smaller nodes
- Better spot availability (more capacity in smaller sizes)
- Burst-arrival batching prevented (S tier: Karpenter can only pick xlarge max)
- Estimated 40-60% cost reduction on idle node waste

## Cost of S/M split specifically

Without S/M split, burst-triggered small pods (33% of runs) batch onto 4xlarge nodes. With variable runtimes (1h vs 8h jobs sharing a node), `WhenEmpty` holds the 4xlarge for the longest job. Example: 3 pods at 2 CPU each with runtimes 1h/1h/8h — combined pool costs 32 xlarge-hours vs S tier costs 10 xlarge-hours (3.2x difference).
